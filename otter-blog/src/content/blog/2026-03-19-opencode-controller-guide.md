# Opencode Controller 使用指南：AI 编码助手的协调控制模式

## 引言

在使用 AI 编码助手时，一个常见的混乱来源是角色不清：什么时候应该自己动手，什么时候交给 AI？什么时候该规划，什么时候该实施？遇到问题时是在编码界面解决还是回到规划界面？

Opencode Controller 技能定义了一套严格的协调控制模式，核心规则很简单：**协调者不写代码，编码者不处理协调。** 本文将解析这套模式的工作流程和设计原则。

## 核心理念

Opencode Controller 不是自己写代码的工具，而是控制 Opencode（AI 编码助手）的工具。职责划分如下：

- **协调者**（本 Agent）：理解用户需求、管理会话、选择 Agent 和模型、协调 Plan 和 Build 的切换。
- **编码者**（Opencode）：在 Plan 模式下分析和规划，在 Build 模式下实施和编码。

这个分离确保了每个角色专注于自己最擅长的事情。

## 预检查

在开始任何工作之前，先完成预检查：

1. 询问用户使用的 AI 提供商（OpenAI、Anthropic、DeepSeek 等）
2. 确认认证方式（API Key、OAuth）
3. 没有确认不继续

预检查的价值在于：避免在配置错误的情况下开始工作。认证失败的调试成本远高于提前确认的成本。

## 会话管理

### 选择已有会话

使用 `/sessions` 命令列出所有可用会话。如果当前项目已有会话，优先复用——不经过用户批准，绝不创建新会话。

这看起来是一个小细节，但它防止了一个常见问题：每次对话都创建新会话，导致历史上下文丢失和会话列表膨胀。

### 创建新会话

只有在以下情况才创建新会话：
- 当前项目没有已有会话
- 用户明确要求创建新会话
- 用户批准创建新会话

## Agent 控制

Opencode 有两个 Agent 模式：

### Plan Agent

负责分析和规划。关键行为：

- 要求 Opencode 分析任务，提供必要的上下文
- 请求清晰的分步计划，每步有明确的输入输出
- 允许 Opencode 提问澄清
- 仔细审查计划，不正确或不完整时要求修改
- 绝不允许在 Plan 中生成代码

Plan 阶段专注于理解和规划。代码生成是 Build 的专属职责。

### Build Agent

负责实施和编码。关键行为：

- 使用 `/agents` 从 Plan 切换到 Build
- 要求 Opencode 按照批准的计划逐步实施
- 遇到问题时立即切回 Plan
- 在 Build 中绝不回答问题

## Plan-Build 循环

这是整个工作流的核心：

```
开始
  ↓
Plan（分析、规划、问答）
  ↓
用户确认计划
  ↓
Build（按计划实施）
  ↓
遇到问题？
  ├─ 是 → 切回 Plan → 回答 → 确认 → 切回 Build
  └─ 否 → 检查结果
      ↓
    用户满意？
      ├─ 否 → 切回 Plan 重新规划
      └─ 是 → 完成
```

### 三个铁律

**绝不跳过 Plan**。每次从 Plan 开始，即使看起来"很显然怎么做"。Plan 是理解的基础，跳过 Plan 就跳过了确认理解的机会。

**Build 中不回答问题**。Build 遇到问题时，必须切回 Plan 处理。在编码时混入规划讨论，既影响编码质量，又影响规划深度。

**用户批准是前提**。创建会话前询问，实施前确认计划，认证前等待确认。没有用户明确同意，不推进。

## 模型选择

使用 `/models` 命令列出可用模型。根据用户偏好选择提供商和模型。

如果需要认证，流程是：Opencode 提供登录链接 -> 发送完整链接给用户 -> 等待用户完成认证 -> 确认后继续。

注意：不要修改或缩短 Opencode 提供的登录链接。完整传递，确保用户能直接使用。

## 输出规范

清晰的输出让用户了解每一步在做什么：

- 显式显示所有斜杠命令（使用代码块或明确格式）
- 说明选择了哪个选项（会话、模型、Agent）
- 逐字提供登录链接

示例：

```
启动 Opencode...
使用 /sessions 选择会话
选择了会话：my-project

使用 /agents 切换到 Plan agent
...
使用 /agents 切换到 Build agent
实施计划中...
```

## 关键洞察

### 职责分离的价值

协调者不写代码，编码者不处理协调。这个分离让每个角色都能专注。协调者专注于理解需求和流程控制，编码者专注于技术实现。

### 循环的必要性

Plan -> Build 的循环不是低效的重复，而是质量的保证。每次循环都是一次理解的校准和实施的重对齐。跳过循环看似节省时间，实际上增加了返工的概率。

### 问题处理的正确流程

Build 中遇到问题切回 Plan，这个规则的本质是"分离关注点"。编码时思考规划会打断编码流，规划时编码细节会干扰规划视野。两个活动各自需要完整的注意力。

## 总结

Opencode Controller 模式看似增加了步骤，实际上通过职责分离和流程纪律提高了整体效率。

Plan-Build 循环确保了"先想清楚再动手"。角色分离确保了"做一件事就专心做"。用户批准确保了"做的就是你想要的"。

对于任何使用 AI 编码助手的场景，这套模式都值得参考：即使是简单的编码任务，一个简短的 Plan 也能避免理解偏差。
