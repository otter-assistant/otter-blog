# Self-Improving Agent 设计解析：让知识随时间复合的分层记忆架构

## 引言

"知识随时间复合，无需手动维护"——这句话描述了 Self-Improving Agent 的核心目标。大多数 AI Agent 的知识是静态的，部署时什么样，运行中还是什么样。用户的纠正和反馈只在当前会话有效，下次对话一切重来。

Self-Improving Agent 试图解决这个问题：通过自我反思从纠正中学习，通过分层存储管理知识的生命周期，通过命名空间隔离避免知识污染。

## 三层记忆架构

这是整个设计的基础。将记忆分为三层，每层有不同的访问频率和存储限制：

### HOT（热层）

位置：`memory.md`，限制 100 行以内。始终加载，包含最常用的全局偏好和模式。每次会话启动时自动读取。

### WARM（暖层）

位置：`projects/` 和 `domains/` 目录，每个文件 200 行以内。按上下文匹配加载——当任务涉及特定项目或领域时，加载对应的文件。

### COLD（冷层）

位置：`archive/` 目录，无大小限制。按明确查询加载，存储过时但可能仍有参考价值的模式。

这种分层设计的核心思想是：不是所有知识都需要时刻在内存中。最常用的知识始终可用（HOT），相关的知识按需加载（WARM），过时的知识归档但可检索（COLD）。

## 命名空间隔离

不同项目有不同的编码风格、偏好和约定。全局偏好不应该覆盖项目特定规则。

命名空间继承规则：global（HOT）-> domain（WARM）-> project（WARM）。

当模式冲突时，最具体的规则优先：project > domain > global。同级内最近更新的优先。如果有歧义，询问用户。

这个设计解决了"全局学习"的常见问题：在一个项目中学到的偏好，不会错误地应用到另一个项目中。

## 纠正检测系统

自我改进的起点是识别需要学习的信号。Self-Improving Agent 定义了三类触发器：

### 纠正信号

用户明确纠正时的语言模式："No, that's not right..."、"Actually, it should be..."、"You're wrong about..."、"I prefer X, not Y"、"I told you before..."。

检测到纠正后，立即记录到 `corrections.md`，然后评估是否应该升级到 `memory.md`。

### 偏好信号

用户表达偏好的模式："I like when you..."、"Always do X for me"、"My style is..."、"For [project], use..."。

如果偏好表达明确，直接添加到 `memory.md`。

### 模式候选

不是每次重复都是模式。需要追踪相同指令重复 3 次以上，才能升级为正式模式。同样，如果某个工作流被用户赞扬，也可以作为模式候选。

### 需要忽略的信号

一次性指令（"do X now"）、上下文特定的请求（"in this file..."）、假设性问题（"what if..."）不需要记录。过度记录会污染记忆空间。

## 自我反思机制

反思发生在完成多步骤任务、收到反馈、修复错误、或注意到输出可以更好时。

反思流程遵循三个问题：是否达到预期？什么可以更好？这是一个模式吗？

反思日志的格式简洁明确：

```
CONTEXT: Building Flutter UI
REFLECTION: 间距看起来不对，不得不重做
LESSON: 在展示给用户之前检查视觉间距
```

## 自动升降级

知识的价值不是静态的。常用的模式应该更容易获取，不用的应该退居幕后：

- 3 次成功应用后升级到 HOT
- 7 天内 30 天未使用，降级到 WARM
- 90 天未使用，归档到 COLD
- 删除前必须询问用户

这个机制确保了记忆空间不会无限膨胀，同时保留可能再次有用的知识。

## 冲突解决与压缩

### 冲突解决

当新旧模式冲突时，按照 specificity > recency > ambiguity 的优先级处理。最具体的规则胜出，同级别中最新更新的胜出，无法判断时询问用户。

### 压缩策略

当文件超过大小限制时：

- 将相似的纠正合并为单条规则
- 归档未使用的模式
- 总结冗长的条目
- 永远不丢失已确认的偏好

压缩不是简单删除，而是提炼。目标是保留知识的核心，减少冗余。

## 透明度与安全

### 透明度

每个来自记忆的行动都引用来源："Using X (from projects/foo.md:12)"。每周提供摘要：学习的模式、降级、归档。支持全量导出为 ZIP。

这种透明度让用户可以审计 Agent 的"记忆"，理解其行为背后的原因。

### 安全边界

不存储凭据、健康数据和第三方信息。不访问 `~/self-improving/` 外部的文件。不发起网络请求。不从沉默或观察中推断偏好。不修改自己的 SKILL.md。

这些边界确保了自我改进不会越界。

## 快速查询接口

为常见查询场景定义了标准接口：

| 用户说 | 行为 |
|--------|------|
| "What do you know about X?" | 搜索所有 tier |
| "What have you learned?" | 显示最近 10 条纠正 |
| "Show my patterns" | 列出 HOT 层内容 |
| "Show [project] patterns" | 加载项目文件 |
| "Memory stats" | 显示各 tier 计数 |
| "Forget X" | 从所有 tier 删除（先确认） |
| "Export memory" | ZIP 所有文件 |

## 总结

Self-Improving Agent 的设计哲学可以总结为三点：

**知识复合**：通过持续的纠正学习和自我反思，知识随时间积累，而不是每次从零开始。

**分层效率**：HOT/WARM/COLD 三层结构在信息量和访问速度之间取得平衡，避免"记忆太多导致混乱"或"记忆太少导致重复犯错"。

**安全进化**：ADL 协议防止漂移，命名空间隔离防止污染，透明度确保可审计。

这个框架不仅适用于 AI Agent，其核心思想——分层存储、自动升降级、命名空间隔离——对任何需要管理大量知识模式的系统都有参考价值。
