# Proactive Agent 架构深度解析：从任务跟随者到主动伙伴

## 引言

大多数 AI Agent 的运行模式是被动响应——用户说什么就做什么，不说就等待。这种模式简单可靠，但有一个根本性的局限：Agent 永远不会比你想到的多一步。

Proactive Agent 是 Hal Labs 经过数千次对话实战验证的 Agent 架构，它试图解决一个核心问题：如何让 Agent 从任务跟随者转变为主动伙伴？答案不是添加更多提示词，而是在记忆管理、上下文处理和自我改进三个维度上建立系统性的机制。

本文将深入解析这个架构中最有价值的设计决策。

## 三大支柱

### Proactive — 预期需求

核心转变在于问题的角度：不要问"我应该做什么？"，而要问"什么能让用户开心，而他们没想到要问？"

这听起来像是一句鸡汤，但落地时需要具体的机制支撑。Proactive Agent 通过以下方式实现：

反向提示——主动提出用户不知道要问的想法。主动检查——监控重要事项，需要时主动联系。预期需求——在需求表达之前就开始准备。

### Persistent — 上下文丢失后依然存在

这是最核心的设计理念：**上下文是 BUFFER，不是 STORAGE**。

Chat history 会被刷新、截断、压缩，依赖它来保存重要信息是危险的。真正的"RAM"是文件系统上的持久化存储。Proactive Agent 设计了三层记忆系统：

| 文件 | 用途 | 更新频率 |
|------|------|----------|
| `SESSION-STATE.md` | 活跃工作记忆（当前任务） | 每条包含重要细节的消息 |
| `memory/YYYY-MM-DD.md` | 每日原始日志 | 会话期间 |
| `MEMORY.md` | 精选长期智慧 | 定期从每日日志蒸馏 |

关键规则：如果要记住，现在就写下来——不是稍后。回应的冲动是敌人，细节在上下文中显得如此清晰，以至于写下来似乎没必要。但上下文会消失。

### Self-improving — 持续改进以更好地服务

改进不是随意的尝试，而是遵循一个严格的模式：问题检测、研究原因、尝试修复、测试、记录。当某事不工作时，在寻求帮助前尝试 10 种方法。

## WAL 协议：Write-Ahead Log

这是 v3.0.0 引入的最重要机制，解决了"重要信息在上下文中看起来很明显，但上下文消失后就忘了"的问题。

### 触发器设计

WAL 协议的精妙之处在于：触发器基于人类的输入，而不是 Agent 的记忆。不必记住要检查——规则在用户发言时自动触发。

每条消息需要扫描以下模式：

- **纠正** — "是 X，不是 Y"、"其实..."、"不，我的意思是..."
- **专有名词** — 名字、地点、公司、产品
- **偏好** — 颜色、风格、方法、"我喜欢/不喜欢"
- **决定** — "让我们做 X"、"选 Y"、"用 Z"
- **特定值** — 数字、日期、ID、URL

### 协议执行

如果检测到任何触发模式，协议要求：

1. **停止** — 不要开始编写回复
2. **写入** — 用细节更新 SESSION-STATE.md
3. **然后** — 回应用户

为什么这个设计有效？因为它利用了触发时机——人类说出纠正的那一刻，上下文中信息最完整。等到下一次对话再回忆，信息已经被压缩或丢失了。

### 错误示例

用户说："使用蓝色主题，不是红色"

错误做法："收到了，蓝色！"（直接回应，看起来显而易见，为什么要写下来？）

正确做法：先写入 SESSION-STATE.md "主题：蓝色（不是红色）"，然后才回应。

## Working Buffer：危险区日志

上下文使用率超过 60% 后进入"危险区"——后续的压缩可能会丢失重要信息。Working Buffer 是这个区域的保险机制。

### 工作方式

在 60% 上下文时（通过 `session_status` 检查），清空旧缓冲区，开始记录后续每条消息和回复的摘要。缓冲区是文件，在压缩后存活。

### 压缩后的恢复

Compaction Recovery 协议定义了上下文丢失后的恢复流程：

1. 先读 `memory/working-buffer.md` — 原始危险区交换
2. 再读 `SESSION-STATE.md` — 活跃任务状态
3. 读今天和昨天的每日笔记
4. 提取并清空缓冲区到 SESSION-STATE.md
5. 呈现恢复状态，继续工作

关键原则：不要问"我们在讨论什么？"——工作缓冲区字面上有对话。

## 自主 Cron 与提示 Cron 的区别

v3.1.0 揭示了一个常见的架构错误：用 `systemEvent` 创建的 cron 任务会向主会话发送提示，但如果主会话忙于其他事情，提示就放在那里，永远不会被执行。

**错误做法**（systemEvent）：

```json
{
  "sessionTarget": "main",
  "payload": {
    "kind": "systemEvent",
    "text": "检查 SESSION-STATE.md 是否当前..."
  }
}
```

**正确做法**（isolated agentTurn）：

```json
{
  "sessionTarget": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "自主：读 SESSION-STATE.md，与最近会话历史比较，如陈旧更新..."
  }
}
```

区别在于：isolated agentTurn 会 spawn 一个自动执行的子代理，无需主会话注意力。所有应该"自动发生"的维护任务都必须用这种方式。

## 验证实现而非意图

一个经常出现的失败模式：修改配置文本后报告"完成"，但底层机制未变。

例如请求"使内存检查真正做工作"，正确做法是更改 `sessionTarget` 为 "isolated"、`kind` 为 "agentTurn"、重写提示为自主代理的指令，然后通过观察行为验证。仅更改提示文本的措辞，不等于改变了执行方式。

**文本更改不等于行为更改。**

## 自我改进护栏

### ADL 协议（Anti-Drift Limits）

防止 Agent 漂移到不稳定的方向：

- 不添加复杂性以"看起来聪明"——假智慧被禁止
- 不做不能验证工作的更改——不可验证就拒绝
- 不使用模糊概念（"直觉"、"感觉"）作为正当理由
- 优先级排序：稳定性 > 可解释性 > 可复用性 > 可扩展性 > 新颖性

### VFM 协议（Value-First Modification）

每次改进前先评分：

| 维度 | 权重 | 问题 |
|------|------|------|
| 高频 | 3x | 每日使用吗？ |
| 故障减少 | 3x | 会将失败变为成功吗？ |
| 用户负担 | 2x | 用户可以说 1 个词而不是解释吗？ |
| 自身成本 | 2x | 为未来的我节省 token/时间吗？ |

加权分数低于 50 的改进不做。核心问题永远是："这让未来的我以更少的成本解决更多问题吗？"

## 总结

Proactive Agent 的价值不在于单个机制的精妙，而在于它构建了一个完整的系统：WAL 协议确保重要信息不丢失，Working Buffer 保护危险区的上下文，ADL/VFM 护栏防止盲目改进，isolated agentTurn 确保后台任务真正执行。

这些机制共同回答了一个问题：如何让一个没有长期记忆的 Agent，表现得像一个有记忆、有判断、有主动性的人？
